# VisionOne â†’ Jira Webhook Integration

Automated security alert pipeline that bridges **Trend Micro Vision One** with **Jira**, creating structured incident tickets from XDR alerts in real time.

---

## How it works

1. Vision One fires a webhook to this service when an alert is triggered
2. The service validates the HMAC signature to confirm the request is legitimate
3. If the payload is summarized, it fetches full alert details from the Vision One API
4. It extracts all impacted entities (hosts, accounts, email addresses, URLs)
5. For each entity, it checks for an existing Jira issue to avoid duplicates
6. If none exists, it opens a new Jira issue with severity, priority, matched rules, and technical details

---

## Requirements

- Python 3.11+
- A [Jira](https://www.atlassian.com/software/jira) project with API access
- A [Trend Micro Vision One](https://www.trendmicro.com/en_us/business/products/detection-response/xdr.html) account with webhook configuration enabled
- A hosting platform that exposes a public HTTPS endpoint (e.g. [Render](https://render.com))

## Running locally

```bash
# Install dependencies
pip install -r requirements.txt

# Copy and fill in environment variables
cp .env.example .env

# Start the server
uvicorn main:app --reload
```

The service will be available at `http://localhost:8000`.  
API docs are auto-generated by FastAPI at `http://localhost:8000/docs`.

## Configuring the Vision One webhook

In Vision One, go to **Workflow and Automation â†’ Third-Party Integration â†’ Webhook** and create a new webhook with:

| Field | Value |
|---|---|
| URL | `https://<your-app>.onrender.com/webhook/visionone` |
| Body | `{"type":"$type$","data":"$payload$"}` |
| Custom header | `X-TrendMicro-Signature: <hmac-sha256-of-body>` |

## Supported entity types

The service creates a Jira issue for each impacted entity of the following types found in the alert's `impactScope`:

- `host`
- `account`
- `emailaddress`
- `url`

Entities of other types are silently ignored.

---

## Duplicate prevention

Before creating an issue, the service queries Jira using the alert ID. If a matching issue already exists in the configured project, the entity is skipped and logged as `[Skipped]`.
